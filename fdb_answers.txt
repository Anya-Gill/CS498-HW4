Q1: FoundationDB separates its Transaction Management System from its Storage System so that each layer can scale and fail independently. The TMS handles transaction coordination such as accepting writes, assigning version numbers, performing conflict checks, and writing to distributed logs while the SS is responsible for persisting key-value data on disk and serving reads. This decoupling means the storage layer can be repartitioned or recovered without affecting in-flight transaction processing, and vice versa.

Q2: The Resolver component is responsible for detecting read-write conflicts between concurrent transactions. This check occurs after the client submits a commit request and the Sequencer assigns a commit version, but before the transaction's mutations are written to the LogServers. The Resolver verifies that no other committed transaction has modified any key in the committing transaction's read set since its read version and if a conflict is found, the transaction is aborted.

Q3: The read version is a logical timestamp that defines the snapshot of the database a transaction will read from, ensuring the transaction sees a consistent, point-in-time view of all committed data up to that version. This is central to FDB's strict serializability guarantee that by pinning all reads to a single version and later checking for conflicts against that same version at commit time, FDB ensures that the transaction behaves as if it executed atomically at one point in the version timeline.

Q4: Standard reads in FDB are added to the transaction's read conflict range, meaning any concurrent write to those keys will cause the transaction to abort at commit time. Snapshot reads bypass this conflict detection as they still observe a consistent view at the transaction's read version, but they do not create conflict ranges. You would choose snapshot reads when you need to read data for informational purposes like monitoring or analytics queries within a transaction where staleness from a concurrent write is acceptable, and you want to avoid unnecessary aborts.

Q5: FDB's recovery avoids replaying a redo log in the critical path because, during normal operation, StorageServers continuously pull committed mutations from the LogServers and apply them to their local persistent storage in the background. When a failure occurs, the recovery process only needs to identify the last durable commit version in the logs and tell the new TMS to start from there; there is no long replay phase because StorageServers have already applied most or all of the committed data. Any small gap between what a StorageServer has applied and the latest durable log entry is caught up by reading the remaining log entries, which is typically very small.
